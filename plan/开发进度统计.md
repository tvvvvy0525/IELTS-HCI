已完成：

- 错题解析和答案解释没有进入主练习页闭环。
- 练习记录虽然写入 `localStorage`，但 History 页面未展示。

### 5.3 Listening 练习闭环

已完成：

- Listening 题库目录浏览
- 按 Part 分类
- iframe 加载原始听力练习页面
- 原始练习页面内完成答题
- 父页面注入 bridge 脚本
- 完成练习后同步成绩到 `localStorage`
- 手动切换对错后可同步更新成绩

当前缺口：

- Listening 依赖 iframe 与原始 HTML 页面，和主应用的 UI/状态管理还没有统一。
- manifest 中路径使用本地绝对路径，部署到其他机器或线上环境时需要重构资源路径。
- 注入脚本依赖原始 HTML 的 DOM 结构，原始页面结构变化会影响同步逻辑。
- History 页面未展示听力记录。

## 9. 当前主要风险

| 风险 | 影响 | 优先级 |
|---|---|---:|
| Listening manifest 使用本地绝对路径 | 影响跨设备运行和线上部署 | 高 |
| History 未展示记录 | 学习闭环不完整 | 高 |
| Dashboard 统计为静态值 | 用户无法感知进度 | 高 |
| 写作和口语仍是占位模块 | 与“全科备考平台”目标差距大 | 高 |
| `generated/` 与 `public/generated/` 存在重复资源 | 仓库体积大，维护成本高 | 中 |
| Reading 批改逻辑偏简单 | 可能误判合法答案 | 中 |
| Tools / Settings 静态化 | 产品完整度不足 | 中 |

## 10. 建议下一步

建议按以下顺序推进：

1. 完成 History 页面，先把 `exam_history` 中的 Reading / Listening 记录展示出来。
2. 完成 Dashboard 数据接入，让首页能显示真实练习次数、正确率和学习时长。
3. 重构 Listening 资源路径，避免依赖本机绝对路径。
4. 给 Reading 批改加入更稳健的答案 normalize 逻辑。
5. 实现 Tools 中的 Band score 换算器，这是小功能但能快速提升完整度。
6. 再进入 Writing 或 Speaking 大模块开发。
