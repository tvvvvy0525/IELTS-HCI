# Ielts-HCI 产品开发流程

## 1. 目标

为 `Ielts-HCI` 建立统一的产品开发节奏，确保每次需求都先做产品判断，再做代码实现，最后做验证与回顾。

## 2. 固定流程

### 阶段 A：需求入口

每次收到新需求，先回答 4 个问题：

1. 这是新功能、体验优化、数据闭环，还是部署问题？
2. 它影响的是 `Reading`、`Listening`，还是占位中的 `Writing/Speaking`？
3. 这次要先出方案，还是已经获批可以直接实施？
4. 完成后如何验证，是否必须经过 `npm run build`？

产出：
- 更新 `plan/task_plan.md`
- 记录需求与约束到 `plan/findings.md`

### 阶段 B：现状盘点

做任何实现前，先确认：

- 当前页面是否已有 UI 壳
- 数据源是否已经存在
- 缺的是“逻辑层”“展示层”还是“资源层”
- 有没有部署风险或历史兼容问题

适用原则：
- 优先补齐已有能力的闭环，不优先扩新模块
- 先确认最短路径价值，再决定是否大改结构

### 阶段 C：里程碑定义

每次只推进一个明确里程碑。每个里程碑必须满足：

- 有清晰的用户价值
- 有明确的完成标准
- 有可执行的验证方式

推荐格式：

```md
Milestone 名称
- 目标：
- 范围：
- 不做：
- 完成标准：
- 验证方式：
```

### 阶段 D：任务拆解

进入开发前，把里程碑拆成 3 类任务：

1. 数据层
例如：统一 `exam_history` 结构、聚合统计、持久化规则。

2. 页面层
例如：`Dashboard` 接真数据、`History` 接筛选与列表、回流入口。

3. 风险层
例如：`Listening` 的绝对路径、iframe bridge 稳定性、构建兼容性。

拆解要求：
- 每个任务都要对应具体文件
- 每个任务都要说明验证方法
- 没有完成标准的任务不能进入实施

### 阶段 E：实施执行

实施时遵循以下规则：

- 先小步改动，再做下一步
- 每完成一个子任务就更新 `plan/progress.md`
- 不同时推进两个高耦合大改
- 结构改动后默认跑 `npm run build`
- 不为占位模块提前设计过重架构

### 阶段 F：验证交付

每轮开发完成后至少检查：

- 路由是否可进入
- 关键页面是否能展示真实数据
- 主要操作是否可回流到下一步
- `npm run build` 是否通过
- 风险项是否写回计划文件

## 3. 当前项目的优先级规则

当前仓库按以下顺序推进：

1. 学习数据闭环
目标是把 `Reading/Listening -> exam_history -> Dashboard/History -> 重新练习` 串起来。

2. Listening 可部署化
目标是减少本机绝对路径和 `/@fs/` 依赖，降低迁移与上线风险。

3. Reading/Listening 体验补齐
目标是提升批改容错、批注持久化、复习入口完整度。

4. Writing/Speaking MVP
目标是先做可用最小版本，而不是一步到位做完整 AI 功能。

## 4. 当前推荐里程碑

### Milestone 1：Reading / Listening 学习闭环

- 目标：
  让用户能看到自己的练习结果、统计变化和复习入口。
- 范围：
  - `Dashboard` 接真实统计
  - `History` 接真实记录
  - 支持按科目筛选
  - 支持从记录重新进入练习
  - 统一 `exam_history` 数据格式
- 不做：
  - Writing / Speaking 新功能
  - 大规模视觉重做
  - 完整 AI 评分
- 完成标准：
  - Dashboard 不再显示静态 0
  - History 能展示至少 Reading / Listening 真实记录
  - 用户可以从记录重新开始一次练习
  - 本地构建通过
- 验证方式：
  - 手动造 2 到 3 条练习记录验证展示
  - 路由回跳验证
  - `npm run build`

## 5. 文档使用约定

- `plan/task_plan.md`：放阶段状态、里程碑、关键决策
- `plan/findings.md`：放现状发现、技术判断、风险
- `plan/progress.md`：放会话记录、测试结果、执行日志

后续每次新需求，都先更新这三份文件，再决定是否进入编码。
