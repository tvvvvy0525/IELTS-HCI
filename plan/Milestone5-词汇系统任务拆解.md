# Milestone 5：词汇系统产品文档

## 1. 文档目标

本文档用于指导组内成员在 `IeltsCoach` 中实现完整的词汇系统模块。目标是让接手者在不了解历史上下文的情况下，也能直接理解：

- 为什么要做这个模块
- 这个模块要解决什么问题
- 需要做出哪些页面、功能和数据结构
- 哪些能力是本轮必须完成的
- 实现时应遵守哪些交互和业务规则
- 如何判断这个模块已经做完

本文档面向产品、设计、前端开发和负责四科联动的同学共同使用。

---

## 2. 模块定位

词汇系统是 `IeltsCoach` 的正式一级模块，不是附属工具页，也不是仅供演示的辅助页面。

它在产品中的角色是：

- 为用户提供可持续积累的个人词库
- 为用户提供低负担、可长期坚持的复习流程
- 为阅读、听力、写作、口语四科提供统一的词汇沉淀层
- 让词汇学习从“孤立背单词”升级为“词义 + 词族 + 来源场景”的结构化学习

对应主产品文档中的定位：

- 词汇系统是 `个人词库 · 间隔复习 · 词族地图`
- 词汇系统承担四科支撑层职责
- 词汇系统需要与首页今日任务看板和四科联动能力打通

## 3. 用户问题

本模块主要解决以下问题：

### 3.1 P3：单词孤立，缺乏联系

当前大多数背词体验只有“单词 - 中文释义”的线性结构。用户背下一个词，不等于能识别它的同义词、派生词或语境变化。

本系统需要让用户在单词详情中直接看到：

- 同义词
- 衍生词
- 词性差异
- 来源场景

### 3.2 P4：学了就忘，不知道何时复习

用户通常不知道今天该复习什么，只能被动等待或靠自觉回忆。

本系统需要提供：

- 今日待复习任务
- 明确的复习入口
- 可持续的复习节奏

### 3.3 P5：传统背词产品容易积压

很多背词产品会不断累计待复习任务，导致用户看到越来越大的任务数字后直接放弃。

本系统必须避免：

- 欠债式任务增长
- 单日任务无限膨胀
- 用户一打开页面就产生心理负担

## 4. 本轮目标

本轮不是做一个“能跑起来”的简化版，而是做一个可以交给多人协作完成的完整词汇系统版本。

### 4.1 本轮必须覆盖的能力

- 基础词库导入与标准化
- 个人词库浏览与搜索
- 单词详情页
- 同义词 / 衍生词关系维护
- 新增词和编辑词
- 今日复习
- 至少两种复习模式
- 词汇统计
- Dashboard 快速入口与最小词汇任务
- 四科联动接口预埋

### 4.2 本轮不追求的能力

- 登录与云同步
- 多人共享词库
- AI 自动扩词
- 发音评分
- 真正的图谱可视化画布
- 复杂 FSRS / SM-2 全量算法

说明：
这些能力不是不重要，而是不适合作为当前 milestone 的交付边界。

## 6. 目标用户与核心使用场景

### 6.1 用户类型

- 自备考生：需要免费、可持续的词汇学习工具
- 练习型用户：阅读 / 听力中遇到生词，希望立刻沉淀
- 复考用户：已经有一定词汇量，但需要更有效率地复习和查漏补缺

### 6.2 核心使用场景

#### 场景 A：打开首页，开始今日词汇任务

用户进入首页后，应能直接看到：

- 今天建议复习多少词
- 当前未掌握词数
- 进入今日复习的 CTA

#### 场景 B：在词库中查词和整理关系

用户进入词库后，应能：

- 搜索单词
- 按主题、词性、状态筛选
- 查看释义、例句、来源
- 添加同义词和衍生词

#### 场景 C：自己录入新词

用户可以手动录入：

- 新单词
- 释义
- 例句
- 同义词
- 衍生词

若关联词已存在于词库，则两边都应显示对方。

#### 场景 D：从四科模块把词沉淀进词库

虽然四科联动由其他同学后续接入，但本模块必须预留统一接口，确保：

- 阅读可加词
- 听力可加词
- 写作可加词
- 口语可加词

## 9. 复习策略

## 9.1 状态模型

建议采用三态：

- `new`
- `learning`
- `known`

原因：

- 如果只有 `unknown / known` 两态，完整系统很难支持后续多模式和调度
- 三态可以兼顾前台简洁和后台可扩展

### 状态含义

- `new`
  - 进入词库但尚未完成有效学习
- `learning`
  - 已接触过，但仍需要重复巩固
- `known`
  - 当前阶段视为已掌握，不优先进入高频复习

## 9.2 今日任务生成规则

- 默认一轮任务量为 20
- 先取到期的 `learning`
- 数量不足时再补 `new`
- 默认不一次性把全部待复习推给用户

## 9.3 反馈规则

### 闪卡模式

- `认识`
  - `new -> learning`
  - `learning -> known`
- `模糊`
  - 保持 `learning`
- `不认识`
  - 保持或回到 `learning`

### 选择题模式

- 答对：
  - 提升掌握度
- 答错：
  - 保持或回退到 `learning`

## 9.4 防积压规则

- 首页不显示“你欠了多少词”
- 只显示：
  - 今日建议任务
  - 当前学习中词数
  - 当前已掌握词数
- 单日任务默认上限为 20
- 超额任务顺延，不一次性全部派发

## 10. 数据结构定义

## 10.1 基础词条结构

每个词条至少应包含：

```js
{
  id: string,
  lemma: string,
  variants: string[],
  pos: string,
  meaning: string,
  example: string,
  extra: string,
  topic: string,
  groupKey: string,
  source: string,
  sourceType: string,
  createdAt: string,
  updatedAt: string
}
```

## 10.2 用户扩展结构

```js
{
  customWords: Record<string, Word>,
  customOverrides: Record<string, Partial<Word>>,
  notesByWordId: Record<string, string>,
  archivedWordIds: string[]
}
```

## 10.3 关系结构

```js
{
  synonymIds: string[],
  derivativeIds: string[],
  pendingRelationTexts: string[]
}
```

说明：

- 词库中已有的关系，优先保存为 `id`
- 暂未入库的关系，先保存为文本

## 10.4 学习记录结构

```js
{
  sessionId: string,
  wordId: string,
  mode: 'flashcard' | 'choice' | 'dictation',
  result: 'known' | 'fuzzy' | 'unknown' | 'correct' | 'wrong',
  reviewedAt: string,
  responseMs: number,
  sourceContext: string
}
```

## 11. 四科联动接口要求

本轮不要求四科全部立即接完，但词汇系统必须提前提供稳定接口，供其他同学调用。

### 11.1 接口目标

- 能从阅读加入词汇
- 能从听力加入词汇
- 能从写作收藏词汇
- 能从口语记录待提升词汇

### 11.2 推荐接口

- `captureReadingWord(payload)`
- `captureListeningWord(payload)`
- `captureWritingWord(payload)`
- `captureSpeakingWord(payload)`

### 11.3 来源标签规范

来源标签建议采用统一格式，例如：

- `Reading-P1`
- `Reading-P2`
- `Listening-Part3`
- `Writing-Task2`
- `Speaking-Part1`

### 11.4 联动后的预期行为

- 词被加入词库
- 自动写入来源标签
- 若已存在，则更新来源信息而不是重复造词
- 可选择进入 `new` 或 `learning`

## 13. 实现要求

### 13.1 资源层要求

- 不允许页面直接依赖外部 repo 的绝对路径
- 必须先把原始词汇资源标准化为当前仓库内部资源
- 必须提供稳定索引能力：
  - lemma 索引
  - variant 索引
  - topic 索引

### 13.2 数据层要求

- 页面组件不应直接拼 `localStorage` 逻辑
- 词库、关系、复习记录、统计应通过统一服务层管理
- 数据服务层必须可测试

### 13.3 交互要求

- 首页 CTA 优先是“开始今日复习”
- 用户常用路径应控制在 1 到 2 次点击内
- 不要出现过多专业术语
- 不要展示让用户产生失败感的积压数字

### 13.4 兼容性要求

- 不得破坏现有 Reading / Listening / Writing / History 主流程
- 词汇模块应独立可访问
- 无词库记录时应有正常空状态
